# M14.4 剪辑端真实剪切流水线设计

## 背景

M14.1 已经把剪辑端主工作台收敛为“素材定位、剪切任务、本地素材、公共素材库、设置”。M14.2 完成了基于文案的搜、选、预览和剪切入口。M14.3 完成了剪切任务自动刷新、完成后本地素材回流、以及本地素材优先参与搜索。

当前真实 API 模式还有一个关键断点：剪辑师点击“剪切这段”后，前端会创建剪切任务并轮询任务状态，但 Cutter API 的真实剪切执行需要调用 `/cutter/cut-jobs/run-next`。如果没有手动触发，任务会停留在等待中，剪辑师会看到“加入任务了，但没有在剪”的体验。

M14.4 的目标是补上这段生产链路，让剪辑端在真实 API 模式下具备最小可用的本机剪切流水线。

## 产品目标

剪辑师不应该理解“run-next”这类系统按钮。剪辑师的动作只有一个：

`拖选文案 -> 剪切这段`

系统随后应该自动完成：

1. 创建剪切清单。
2. 提交剪切任务。
3. 启动本机剪切流水线。
4. 按顺序执行等待中的剪切任务。
5. 刷新任务状态。
6. 剪切完成后刷新本地素材库。
7. 新本地素材参与后续搜索。

剪辑师可以继续停留在素材定位页继续搜、选、剪，不需要跳到剪切任务页守着进度。

## 范围

本轮做：

- API 模式下，提交剪切任务后自动运行本机剪切流水线。
- 流水线一次执行一个任务，直到队列没有等待任务。
- 剪切任务页展示流水线状态：空闲、运行中、最近完成、失败。
- 素材定位页和剪切任务页继续自动刷新队列。
- 任务完成后刷新本地素材，沿用 M14.3 的本地素材回流逻辑。
- 失败任务保留在队列中，显示失败；流水线继续尝试后续等待任务。
- 所有新增用户可见文案使用中文。

本轮不做：

- 不做多并发剪切。
- 不做任务暂停、取消、优先级、批量删除。
- 不做后台常驻 worker 或独立系统服务。
- 不做 Tauri 原生打开文件夹能力。
- 不做公共素材下载同步到本地。

## 推荐方案

采用“前端驱动的本机剪切流水线”。

前端在 CutterApp 中拥有剪切任务的用户上下文和页面状态。提交任务成功后，前端调用 Cutter API 的 `runNextCutJob()`，等接口返回后刷新任务列表和本地素材。只要任务列表仍有等待中任务，就继续运行下一条。

这个方案的优势：

- 能快速闭合真实剪切体验。
- 不需要引入新的后台守护进程和生命周期管理。
- 与当前 Cutter API 已有能力匹配。
- 失败时可以直接在剪辑端反馈中文状态。

这个方案的边界：

- 页面关闭后流水线不会继续运行。
- 长时间剪切期间前端请求会等待当前任务完成。
- 这是 M14 阶段的最小可用执行闭环，后续可升级为服务端后台 worker。

## 状态模型

新增剪切流水线状态：

- `idle`：空闲，没有正在由前端驱动的运行。
- `running`：正在调用 Cutter API 执行等待任务。
- `completed`：最近一次流水线运行结束，至少处理过一个任务。
- `failed`：最近一次 run-next 调用失败。

状态展示中文：

- 空闲：`本机剪切空闲`
- 运行中：`本机剪切运行中`
- 完成：`本机剪切已完成`
- 失败：`本机剪切失败`

任务列表仍然以单个任务状态为准：

- `pending`：等待中。
- `running`：剪切中。
- `done`：已完成。
- `failed`：失败。
- `cancelled`：已取消。

## 数据流

### 提交剪切

1. `MaterialLocatorPage` 的“剪切这段”触发 `CutterApp`。
2. `CutterApp` 调用 `createClipList()`。
3. `CutterApp` 调用 `submitCutJobs()`。
4. 提交成功后设置 `hasSubmittedCutJobs = true`。
5. 提交成功后启动本机剪切流水线。

### 运行流水线

1. 流水线开始后设置状态为 `running`。
2. 调用 `runNextCutJob()`。
3. 如果返回 `null`，表示没有等待任务，流水线结束。
4. 如果返回 `done`，刷新队列和本地素材。
5. 如果返回 `failed`，刷新队列，继续尝试下一条等待任务。
6. 每轮最多连续执行一个受控上限，避免异常情况下无限循环。
7. 流水线结束后设置 `completed` 或 `idle`。

### 自动刷新

M14.3 已经有队列自动刷新规则。M14.4 保留该规则，并补充：

- 流水线运行中仍显示自动刷新状态。
- run-next 返回后立即刷新队列。
- 队列出现新的完成任务时刷新本地素材。

## 错误处理

- `runNextCutJob()` 请求失败：显示 `本机剪切失败` 和中文错误，保留队列现状。
- 单个任务失败：任务状态为失败，流水线继续处理后续等待任务。
- 本地素材刷新失败：显示中文错误，但不回滚已完成任务。
- API 未配置 workspace：接口错误直接显示为中文错误，提示需要设置本地工作区。

## 测试策略

单元测试：

- 流水线状态 helper 能根据任务和运行状态生成中文标签。
- 流水线运行决策不会在 fixture 模式、未登录、非剪切相关页面误触发。
- API client 的 `runNextCutJob()` 仍然带认证头。

应用测试：

- `CutQueuePage` 渲染流水线中文状态。
- `CutterApp` 在 API 模式提交任务后会触发 run-next。
- run-next 完成后会刷新剪切队列和本地素材。
- run-next 失败时保留页面并显示中文错误。

浏览器烟测：

- 启动 Cutter API 和 Cutter Web。
- 在素材定位页选中文案并点击“剪切这段”。
- 等待任务从等待中进入已完成。
- 打开本地素材库，确认新增本地素材可见。
- 回到素材定位搜索被剪文案，确认本地素材分组优先出现。

## 验收标准

1. 真实 API 模式下点击“剪切这段”后，不需要手动点击运行任务，任务可以自动执行。
2. 剪切任务页能看到本机剪切流水线状态。
3. 已完成任务会生成本地素材。
4. 新本地素材进入素材定位搜索，仍然排在公共原素材之前。
5. 单个任务失败时不阻塞后续任务处理。
6. 所有新增用户可见文案为中文。
7. `npm test`、`npm run typecheck`、`npm run build:cutter-web` 通过。
