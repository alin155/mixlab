# M15 剪辑端真实验收补丁设计

## 背景

M13 到 M14.5 已经跑通剪辑端真实主链路：登录审核、读取管理端发布素材、搜索文案、选择文案、真实 FFmpeg 剪切、生成本地素材，并让本地素材参与后续搜索。

M15 不继续扩展大功能。它的目标是把这条链路从“能跑通”打磨到“能被真实测试、能看懂状态、能处理常见失败”。剪辑师和测试者不应该遇到按钮没反应、失败原因藏起来、任务卡住不知道怎么恢复的问题。

## 产品目标

1. 剪切任务失败后，页面必须说明失败原因，并提供真实可用的重试入口。
2. 剪切任务页不再出现无后端能力的假控件；能点击的按钮必须有实际行为。
3. 素材定位页继续保持主路径清晰：选文案后剪切，失败或成功都在当前页面给出中文反馈。
4. 真实 API 模式下，重试失败任务后可以重新进入等待队列，并由本机剪切流水线继续执行。
5. 验收脚本继续证明公共素材到本地素材复用的真实闭环，不污染用户真实素材库。

## 范围

本轮做：

- `cutter-local` 增加失败剪切任务重试能力：只允许重试 `failed` 任务，重试后状态回到 `pending`，清除失败原因和上次执行时间。
- Cutter API 增加受登录保护的重试接口。
- 剪辑端 API client 接入重试接口。
- 剪切任务页展示每个任务的选中文案；失败任务展示失败原因。
- 剪切任务页的“重试”按钮只在有真实 handler 时出现，点击后重新入队并启动本机剪切流水线。
- 素材定位页和剪切任务页继续使用中文，不暴露底层状态枚举、内部路径或英文错误作为主要体验。
- 更新烟测，让端到端链路继续覆盖真实 API 模式、真实 FFmpeg 剪切和本地素材优先。

本轮不做：

- 不做公共素材下载或同步到本地。
- 不做本地视频导入和本地预处理。
- 不做桌面端打包。
- 不做后台常驻 worker。
- 不做任务暂停、取消、优先级、多并发或批量操作。

## 设计方案

采用“小闭环修补”方案：保留当前前端驱动本机剪切流水线，只补齐失败恢复和状态可见性。

失败任务本质上仍然保留完整剪切输入：来源视频、起止时间、选中文案、剪切模式。重试时不需要重新选择文案，也不需要复制一条新任务；系统把同一条失败任务重新置为等待中，由已有 `run-next` 流水线继续执行。这样测试者看到的是同一个任务从失败回到等待，再变为剪切中或完成，语义最清楚。

## 数据流

### 重试失败任务

1. 剪辑任务执行失败，任务状态为 `failed`，带 `error_message`。
2. 剪切任务页展示失败原因和“重试”按钮。
3. 用户点击“重试”。
4. 剪辑端调用 `POST /cutter/cut-jobs/:cut_job_id/retry`。
5. Cutter API 校验剪辑师会话和本地工作区。
6. `cutter-local` 读取任务，确认状态为 `failed`。
7. 系统把任务写回 `pending`，清除 `error_message`、`started_at`、`finished_at`、`export_clip_id`、`output_file`。
8. 剪辑端刷新任务列表，并启动本机剪切流水线。

### 页面表达

剪切任务页每条任务至少展示：

- 中文状态。
- 素材标题。
- 时间范围。
- 选中文案摘要。
- 进度。
- 失败原因，只有失败任务展示。
- 重试按钮，只有失败任务且有 handler 时展示。

## 错误处理

- 重试不存在任务：返回中文错误，页面保持原状态。
- 重试非失败任务：返回中文错误，页面提示该任务当前不需要重试。
- 重试接口失败：页面显示中文错误，不清空当前任务列表。
- 重试成功但再次剪切失败：任务继续显示失败原因，可以再次重试。

## 验收标准

1. 失败剪切任务能看到失败原因。
2. “重试”按钮不是假按钮，点击后真实调用 API。
3. 只有失败任务出现“重试”。
4. 非失败任务不能通过 API 被错误重试。
5. 重试后的任务状态为等待中，并清除旧失败原因。
6. 重试后剪辑端刷新任务列表并启动本机剪切流水线。
7. 页面新增用户可见文案均为中文。
8. `npm test`、`npm run typecheck`、`npm run build:cutter-web`、`npm run smoke:cutter-api-web` 通过。
