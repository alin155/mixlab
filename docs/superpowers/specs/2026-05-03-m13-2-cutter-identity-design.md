# M13.2 剪辑师身份与登录审核设计

## 结论

剪辑师身份不使用“用户名 + IP”识别。M13.2 使用“用户名身份 + 本机设备令牌 + 会话令牌”的三层身份模型：

- 用户名身份：剪辑师主动输入，管理端审核的是这个人和这次设备申请。
- 本机设备令牌：剪辑端首次打开时生成并保存在浏览器本地，用来识别这台剪辑工作台。
- 会话令牌：管理端审核通过后由后端签发，剪辑端保存后自动登录。
- IP 地址：只作为审计和诊断信息记录，不参与登录判断。

这样做的原因是 IP 不稳定，移动硬盘、热点、公司网络、局域网 NAT、路由器重启都会改变或共享 IP。真正稳定且可控的是本机持久设备令牌和后端签发会话。

## 用户路径

1. 剪辑师首次打开剪辑端。
2. 系统生成本机设备身份，并展示清晰的中文设备名。
3. 剪辑师输入用户名提交申请。
4. 管理端“剪辑师用户”页面显示待审核用户、设备名、设备编号、申请时间、最近登录和使用统计。
5. 管理员通过申请。
6. 剪辑端轮询到审核通过，保存会话令牌并进入工作台。
7. 后续打开剪辑端时，系统用本机设备令牌和会话令牌自动登录。
8. 管理员停用用户后，后端撤销会话，剪辑端下次校验时回到申请状态。

## 数据边界

- 后端存储用户、设备和会话，不把会话令牌返回给管理端。
- 剪辑端只保存本机自己的设备令牌和会话令牌。
- 管理端展示设备编号的短标识，避免长 token 挤占界面。
- 使用统计继续绑定 `user_id`，同时保留 `username` 和 `device_id`，便于后续按用户和设备维度分析。

## M13.2 落地范围

- 剪辑端登录门展示身份说明和当前设备名。
- 剪辑端生成更友好的设备名，不再把完整浏览器 User-Agent 直接展示给管理员。
- Cutter API 登录申请记录设备审计信息：最近 IP 和浏览器标识，但不使用它们做身份判断。
- 管理端剪辑师用户页补充身份模型说明、设备编号短标识、申请时间、最近登录和设备审计信息。
- 测试覆盖：设备名生成、登录申请审计字段、管理端用户页中文可读性。

## 非目标

- 不做密码登录。
- 不做短信、邮箱、第三方 OAuth。
- 不把 IP 当成权限条件。
- 不做管理员细粒度角色权限。
