# M12.3 真实预处理流水线稳定性与控制闭环设计

## 背景

M12.1 收敛了管理端主路径，M12.2 让管理员能看懂预处理队列的整体状态。但真实长跑预处理仍然需要更强的控制闭环：启动后要持续处理队列，当前视频要展示真实阶段，失败不能阻塞后续视频，暂停要安全地停在视频边界，页面按钮也要符合当前运行状态。

## 目标

- 预处理任务在真实执行时写回当前阶段，而不是只靠 `processing` 状态推断。
- 预处理页展示“提取音频、上传音频、语音识别、写入文案、生成关键帧、自动发布索引”等真实阶段。
- 启动按钮只在流水线未运行时可用，暂停按钮只在运行或停止中可用。
- 暂停采用安全边界：不打断当前视频内部的 FFmpeg 或 ASR 调用，当前视频完成后停止继续领取新视频。
- 单个视频失败只标记该视频失败，并继续处理后续队列。
- 后端 API 继续保持中文可读，不把内部日志路径、worker id 或协议状态直接暴露给用户界面。

## 非目标

- 不在 M12.3 做多进程并行调度。
- 不在 M12.3 强杀正在运行的 FFmpeg 或 ASR 请求。
- 不改变 DashScope 临时上传方案。
- 不重做剪辑端。

## 数据与状态设计

`preprocess-job.json` 增加两个运行态字段：

- `current_stage`: 当前正在执行的阶段，例如 `extract-audio`、`upload-audio`、`asr`、`write-transcript`。
- `stage_updated_at`: 当前阶段更新时间。

失败时继续保留 `error_stage` 和 `error_message`。管理端展示规则为：

- 正在处理：优先展示 `current_stage`。
- 处理失败：展示 `error_stage`。
- 等待处理：展示“等待处理”和排队估算。
- 已完成：展示发布阶段。

## 执行流程

1. 启动预处理流水线。
2. 初始化素材库。
3. 扫描素材来源。
4. 自动把未处理视频加入队列。
5. 循环领取队列视频。
6. 每个视频执行阶段并写回状态：媒体探测、提取音频、上传音频、语音识别、写入文案。
7. 成功后进入待发布索引。
8. 自动生成封面和关键帧并发布为剪辑师可见。
9. 如果收到暂停请求，当前视频完成后不再领取下一条。
10. 如果单个视频失败，记录失败并继续下一条。

## 用户体验

预处理页保持一个核心生产面板：

- 顶部生产状态告诉管理员是否运行、是否排队、是否需要处理。
- 当前处理视频区域展示当前阶段、阶段进度、已用时、预计剩余和预计完成。
- 队列区域展示排队位置、预计开始和预计完成。
- 右侧任务控制展示运行负荷、服务状态、上次处理结果。
- 启动/暂停按钮随状态自动启用或禁用。

## 验收标准

- API 测试能证明阶段写回会影响 `/api/admin/preprocess/jobs` 的 `stage_label` 和进度。
- Worker 测试能证明每个真实执行阶段都会回写。
- Pipeline 测试能证明暂停请求会在当前视频完成后停止继续领取。
- Pipeline 测试能证明失败视频不会阻塞后续视频。
- 前端测试能证明启动/暂停按钮不会在错误状态下同时可操作。
- `npm run typecheck`、`npm test` 通过。
