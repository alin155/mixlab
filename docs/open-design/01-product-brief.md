# MixLab Open Design 产品 Brief

本文档面向 Open Design，用于生成 MixLab Web App UI 原型。请把它理解为产品和设计 brief，不是工程实现说明。

## 产品一句话定位

MixLab 是面向剪辑团队的本地 / 局域网素材搜索与剪切工作台，让管理员把真实公共原视频变成可搜索素材，让剪辑师按项目快速定位文案、验证画面并生成本地剪切片段。

## 产品详细介绍

MixLab 由管理端和剪辑端组成：

- 管理端负责公共素材库的生产管理：素材来源、扫描、预处理、语音识别、封面和关键帧、索引发布、系统检查、运行配置和剪辑师用户审核。
- 剪辑端负责剪辑师的日常工作：选择或新建项目，搜索公共原素材和本地素材，查看完整文案，选择连续文案或精确文本范围，预览画面，提交剪切任务，复用本地素材。

系统部署在本地或局域网环境，公共素材库通常位于 NAS 或挂载盘，本地剪切产出保存在剪辑师工作区。设计必须尊重公共素材库和本地工作区的边界：管理端可管理公共库，剪辑端不写公共库。

## 目标用户

- 管理员：非纯技术用户，但需要理解素材生产状态、队列、失败、索引发布和系统检查。
- 剪辑师：高频使用者，需要极快搜索、浏览、选段、剪切和复用素材。
- 运维人员：低频使用者，需要排查路径、磁盘、工具、ASR、索引和服务状态。
- 需要确认：用户是否全部使用中文界面；是否存在英文版、权限分级或多团队空间。

## 用户痛点

- 原视频数量大，人工翻找素材慢，无法按文案快速定位片段。
- 公共素材预处理链路复杂，管理员难以判断哪里卡住：未处理、排队、处理中、失败、待发布、可搜索。
- 剪辑师需要同时确认文案命中和画面内容，普通搜索列表无法直接完成剪切决策。
- 剪辑任务、项目、本地素材容易混乱，需要按项目归档。
- NAS、本机工作区、FFmpeg、ASR、索引等运行环境容易出错，但普通用户不理解技术术语。
- 页面数据量大，首屏慢或整页刷新会严重影响信任感。

## 核心价值

- 把大量原视频转化为可搜索、可定位、可剪切的公共素材库。
- 让剪辑师用文案直接定位素材中的可用片段，缩短从搜索到剪切的路径。
- 让管理员用极少步骤判断素材生产链路是否健康，并处理异常。
- 保护公共素材库：剪辑端只读公共库，所有剪切产出进入本地工作区。
- 支持项目化工作流：搜索、剪切任务、本地素材围绕项目组织。

## 主要使用场景

1. 管理员首次配置或检查公共素材库。
2. 管理员扫描新增素材并启动预处理。
3. 管理员监控预处理流水线，处理失败或停滞任务。
4. 管理员确认索引发布后，剪辑师可搜索新素材。
5. 管理员审核剪辑师登录申请。
6. 剪辑师创建项目并开始搜索。
7. 剪辑师在素材搜索页完成“搜索 - 选中素材 - 验证画面 - 选择文案 - 剪切”。
8. 剪辑师查看剪切任务状态并处理失败任务。
9. 剪辑师复用本地素材。

## 核心用户流程

### 管理端核心流程

1. 打开仪表盘。
2. 查看公共素材库状态、可搜索素材、预处理状态、风险提示。
3. 如有新增素材，进入原视频管理或直接执行下一步建议。
4. 启动预处理，观察流水线阶段和任务队列。
5. 对失败任务进行重试，对停滞任务恢复到队列。
6. 确认索引发布，确保剪辑端可搜索。
7. 需要时进入系统检查或设置处理运行环境问题。

### 剪辑端核心流程

1. 剪辑师登录或提交使用申请。
2. 进入首页，新建或选择项目。
3. 在项目内输入关键词或粘贴文案开始搜索。
4. 在候选素材中选择最相关的公共原素材或本地素材。
5. 查看完整文案与视频预览，定位命中处。
6. 点击单句、拖拽多句或选择精确文本范围。
7. 预览选区，确认时间段和画面。
8. 提交剪切任务。
9. 在剪切任务页查看进度，在本地素材库复用完成片段。

## 产品复杂度说明

MixLab 不是单纯的后台列表，也不是普通素材浏览器。它同时具有：

- 双端产品结构：管理端 + 剪辑端。
- 真实文件系统/NAS 数据：路径、磁盘、权限、网络延迟都会影响体验。
- 后台生产流水线：扫描、音频提取、ASR、文案产物、封面、关键帧、索引发布。
- 搜索和剪切主链路：关键词匹配、完整文案、时间段、视频预览、FFmpeg 剪切。
- 大数据量：公共素材和文案段落都可能很多，需要分页、渐进加载和虚拟滚动。
- 权限与边界：管理端可写公共库，剪辑端只读公共库并写本地工作区。
- 运行环境诊断：ASR、FFmpeg、路径、索引、服务状态需要被用户理解。

## 设计目标

- 现代 SaaS Web App，专业、高效、清晰。
- 工具型产品气质，类似 Linear / Notion / Cursor / Vercel 的克制、准确、稳定感。
- 不做传统后台管理系统，不用拥挤的“按钮墙”和大面积表单堆叠。
- 管理端突出“当前是否能服务剪辑团队”和“下一步做什么”。
- 剪辑端突出“项目内快速搜索、验证、选段、剪切”。
- 页面首屏加载目标：尽量 1 秒内可交互；大列表必须分页或渐进加载。
- 每个控件局部刷新，避免整页闪烁。
- 低频和危险动作放入详情、设置、确认弹窗或二级区域。
- 把工程术语翻译成用户语言，必要技术细节只在诊断/详情里出现。

## 设计时最重要的优先级

1. 真实工作流效率：剪辑师从搜索到剪切的路径必须短，管理员从异常到处理的路径必须清楚。
2. 信息层级：一眼能看出当前状态、主要问题、下一步操作。
3. 稳定可信：加载、局部刷新、处理中、错误状态都要可预期，不要闪烁和跳动。
4. 数据密度：这是工具产品，允许高密度，但必须有分组、留白和扫描路径。
5. 安全边界：剪辑端只读公共库；管理端危险动作需要确认，但不要把“安全开关”当作主功能。
6. 大数据性能：列表、文案、搜索结果必须支持分页/虚拟化/渐进加载的设计表达。
7. 中文可读性：按钮、状态、错误和空状态要像真实产品文案，不要工程日志感。
