# ACC-008 / ACC-009 Evidence Drafts

Generate a draft JSON file before each real target run:

```sh
npm run template:windows-evidence -- docs/acceptance/evidence/windows-acc-008.json
npm run template:nas-evidence -- docs/acceptance/evidence/nas-acc-009.json
```

The generated files are intentionally incomplete. Empty strings, `false` checks, and zero counts must be replaced with real screenshots, diagnostics, path-matrix results, and multi-editor run results from the target environment.

Build a standalone target-machine kit from the repository:

```sh
npm run package:evidence-kit
```

You can also download the `mixlab-target-evidence-kit` artifact from the `Target Evidence Kit` GitHub Actions workflow, from the `Cutter Desktop Windows Package` workflow when collecting ACC-008 evidence beside a fresh installer, or from the `Build Admin Docker Images` workflow when collecting ACC-009 evidence beside freshly pushed NAS images.

The kit includes `MANIFEST.json` with SHA-256 digests, byte sizes, and executable flags for every packaged file except the manifest itself. Run `npm run validate:evidence-kit-manifest` and `npm run validate:evidence-kit-drafts` after packaging, or run both commands with `-- /path/to/mixlab-evidence-kit` after downloading an artifact, and keep the manifest with the downloaded artifact so testers can confirm the copied Windows and NAS target folders still contain the expected collectors, self-checks, README files, draft JSONs, and CI-prefilled installer/image metadata. On target machines without the source repository, run the packaged root self-check instead: `powershell -ExecutionPolicy Bypass -File .\evidence-kit-manifest-self-check.ps1 -KitDir .` on Windows, or `sh ./evidence-kit-manifest-self-check.sh .` on NAS/Unix.

Copy `dist/acceptance/mixlab-evidence-kit/windows/` (or the artifact's `windows/` folder) to each fresh Windows target and `dist/acceptance/mixlab-evidence-kit/nas/` (or the artifact's `nas/` folder) to the NAS shell host.

This static check confirms the expected Compose services, public-library mount, worker command, count refresh default, and blank `.env.example` ASR key; it does not replace `docker compose config`, image pull/build checks, or real NAS target evidence.

After target collection, copy the completed evidence package back under `docs/acceptance/evidence/` with attachment folders intact: `windows-acc-008.json` plus `screenshots/`, and `nas-acc-009.json` plus `evidence/`. Collectors are conservative. They only fill what can be detected or explicitly supplied on the target machine; remaining fields must still be completed with real acceptance evidence. Each completed JSON must include `artifact_provenance.repository_commit_sha` as a 40-character commit SHA, `evidence_kit_artifact_name: "mixlab-target-evidence-kit"`, and the official `https://github.com/alin155/mixlab` GitHub Actions run URL that produced the evidence kit. When `npm run package:evidence-kit` runs inside GitHub Actions, the generated draft JSONs prefill provenance from `GITHUB_SHA` and the current run URL; the Windows packaging workflow also prefills installer file name, installer SHA-256, installer version, and installer run URL, and the Admin Docker workflow prefills the NAS image tag. Windows evidence must also include installer version, installer SHA-256, and installer workflow run URL from `https://github.com/alin155/mixlab`; NAS evidence must keep `deployment.image_tag` equal to the same commit SHA, record `deployment.preprocess_count_refresh_interval` from the deployed worker configuration, and include `admin_source_videos_ui` proof that the deployed NAS admin source-videos page shows the real source path, a concrete source video id, load-more row growth, query API response plus rendered result, and ready-filter API response with only ready rows. The final combined gate rejects evidence when ACC-008 and ACC-009 do not reference the same repository commit. The NAS collector preserves prefilled provenance, image tag, and existing `admin_source_videos_ui` values from an existing `nas-acc-009.json` when the matching environment variables are not provided on the NAS shell host. The Windows collector copies supplied screenshots into `screenshots/<os>/` beside `windows-acc-008.json`. When `-InstallerFilePath` is supplied, it records `installer.file_sha256` using `Get-FileHash -Algorithm SHA256`. When `-EvidenceDir .\captures` is supplied, it also auto-picks `doctor-pass`, `engine-status`, `material-locator`, `cut-job` or `completed-cut`, `local-library`, `success-diagnostics.txt`, and `failure-diagnostics.txt` from that folder when the individual screenshot or diagnostics parameters are omitted. Supplied Windows diagnostics are rejected when empty or unless they include the required `stage`, `api`, `log`, `public`, `workspace`, `ffmpeg`, `ffprobe`, `doctor`, and `retry` terms needed by the final gate. Use `-AllPublicLibraryPathsPassed` and `-AllFailureCasesPassed` only after every matrix row has been verified; use `-RequireCurrentEnvironmentComplete` on each Windows 10/11 target pass once captures and matrix rows are ready because it fails before writing incomplete current-environment evidence. The NAS collector copies supplied screenshots/logs/reports into `evidence/` beside `nas-acc-009.json`; if no `worker.log` is supplied, it captures the last 200 lines from `docker compose logs admin-worker`. The worker log must include `count_refresh_interval` matching `deployment.preprocess_count_refresh_interval`. Both collectors write forward-slash relative paths so the final gate can validate the same evidence package on macOS/Linux CI. Empty attachments are rejected during collection. Screenshot attachments are rejected when their `.png`, `.jpg`, `.jpeg`, or `.webp` extension does not match the file signature; the final combined gate also rejects screenshots smaller than 640x360. Text diagnostics, logs, JSON reports, and other text attachments are rejected when they contain ASR keys, signed URLs, full private transcripts, pasted search text, or non-redacted bearer tokens. The final NAS multi-user report must prove at least 2,000 indexed source videos, at least 48,000 indexed transcript segments, `searchd_health_source_video_count` equal to `indexed_source_video_count`, and `searchd_health_segment_count` equal to `indexed_transcript_segment_count`. On NAS hosts with Node or Python 3 available, `.json` attachments are parsed before copying.

For Windows evidence, `installer.file_name` must include `installer.version` so the recorded installer name, version, and SHA-256 describe one artifact.

The NAS kit includes `captures/50-editor-report.json` as an intentionally failing report draft with the required 50 per-editor rows. Replace every draft status, zero metric, empty id, false top-level pass flag, false `location_verified` flag, and false closed-loop flag with real NAS/SMB target run values before running `nas-acc-009-collector.sh`; when Node is available on the NAS host, the collector rejects weak reports before copying them into `evidence/50-editor-report.json`. Run `nas-50-editor-report-self-check.sh` against `captures/50-editor-report.json` before the collector so the raw report can be fixed while the target run is still available. The final report must include top-level `all_searches_passed`, `all_cuts_written_to_local_workspaces`, `public_library_not_written_by_cutters`, and `no_cross_workspace_outputs` values set to true, `searchd_health_index_version` equal to top-level `search_index_version`, `searchd_health_source_video_count` equal to `indexed_source_video_count`, `searchd_health_segment_count` equal to `indexed_transcript_segment_count`, `metrics.usage.search_failure_count` equal to `0`, plus per-editor integer search hit character positions (`search_result_begin_char` / `search_result_end_char`) mapped into the selected full-transcript character offsets, and must not include target-machine run root fields such as `library_root`, `workspace_root`, or `searchd_cache_root`. When running the repository-based 50-editor searchd smoke as part of the NAS target run, prefer `npm run smoke:searchd-nas-rehearsal`: it defaults to the final-gate scale, writes `captures/50-editor-report.json`, and rejects lower-scale overrides. Set `MIXLAB_SEARCHD_NAS_REHEARSAL_RUN_ROOT=/path/to/target-mounted-run-root` when the generated library/workspace/cache directories should live under a NAS or SMB-mounted acceptance folder instead of the system temp directory.

When `node` is available on the collector host and `captures/50-editor-report.json` exists, `nas-acc-009-collector.sh` derives `editor_session_count`, `all_searches_passed`, `all_cuts_written_to_local_workspaces`, `public_library_not_written_by_cutters`, and `no_cross_workspace_outputs` from the report. Validate collected evidence with:

```sh
npm run audit:target-evidence-readiness
npm run validate:windows-evidence -- docs/acceptance/evidence/windows-acc-008.json
npm run validate:nas-evidence -- docs/acceptance/evidence/nas-acc-009.json
```

`npm run audit:target-evidence-readiness` is a short repository-side status report for returned target evidence. It intentionally exits nonzero until both target evidence packages are present, readable, accepted, and tied to the same repository commit; read its `next_actions` field before rerunning a target collector or the final gate.

These individual validators reject non-portable attachment references before the final combined gate: paths must be relative to the evidence JSON, use forward slashes, stay inside the evidence directory, and use field-matching filename stems such as `doctor-pass`, `engine-status`, `admin-web`, `worker`, and `50-editor-report`. They also verify every referenced attachment exists beside that JSON, reject empty files, reject fake or undersized screenshots, parse JSON attachments, scan text attachments for forbidden private data, and deep-check the NAS worker log plus 50-editor report.

When both target files have been completed from real machines, run the final combined gate:

```sh
npm run validate:target-evidence -- docs/acceptance/evidence/windows-acc-008.json docs/acceptance/evidence/nas-acc-009.json
```

This command must fail until both real target evidence files exist, pass their individual validators, and prove artifact provenance from the same repository commit. It reuses the same attachment checks as the individual validators. Attachment paths must be relative to the JSON file directory and use forward slashes; absolute paths, URLs, backslash separators, and `..` traversal are rejected. Attachment filenames must match their evidence fields, such as `doctor-pass`, `engine-status`, `material-locator`, `cut-job`, `local-library`, `admin-web`, `current-json`, `smb-permissions`, `worker`, and `50-editor-report`. Screenshot attachments must be real `.png`, `.jpg`, `.jpeg`, or `.webp` files with matching file signatures and readable dimensions of at least 640x360; `.json` attachments must parse as JSON. The NAS `multi_user_search_cut_report` JSON must prove `status: "passed"`, top-level pass flags for search, local-workspace cuts, public-library write protection, and cross-workspace isolation, at least 50 editors, at least 50 distinct `source_video_id` values, at least 5 distinct `search_query` values with `search_query_count` and `search_queries` matching the per-editor queries, at least 2,000 indexed source videos, at least 48,000 indexed transcript segments, `searchd_health_index_version` equal to top-level `search_index_version`, `searchd_health_source_video_count` equal to `indexed_source_video_count`, `searchd_health_segment_count` equal to `indexed_transcript_segment_count`, positive `search_sla_ms`, `detail_sla_ms`, and `cut_sla_ms` values no higher than 1500, complete search/detail/cut latency summaries with `min_ms <= p50_ms <= p95_ms <= max_ms`, each search/detail/cut `p95_ms` no higher than its matching SLA field, `editor_count` and `active_user_count` equal to `editor_sessions.length`, `distinct_source_video_count` equal to the unique `source_video_id` values in `editor_sessions`, each latency `count` equal to `editor_sessions.length`, each latency `max_ms` at least the maximum matching per-editor timing, usage counts for search, full transcript views, selections, cut submissions, successful cuts, and local clips each at least `editor_sessions.length`, and `search_failure_count` equal to `0`. It must also include `editor_sessions` with at least 50 unique `user_id` values and at least 50 unique `workspace_id` values; each session must prove `search_backend: "searchd"`, `search_index_version` equal to the top-level `search_index_version`, `search_query`, matching `search_result_source_video_id`, `search_result_segment_id`, `search_result_rank` inside `search_result_group_count`, `search_result_group_count` at least `search_result_limit`, `search_result_limit` at least 50, `full_transcript_source_video_id` equal to `source_video_id`, `full_transcript_segment_id`, `full_transcript_segment_count` at least 4, `full_transcript_char_count` greater than `selected_text_char_count`, `full_transcript_text_sha256`, integer `search_result_begin_char` / `search_result_end_char` values inside the full-transcript character offsets, `search_result_text_sha256` equal to `sha256(search_query)`, integer `selected_text_begin_char` / `selected_text_end_char` values inside the full transcript that include the search hit range, `selected_text_char_count` equal to selected range length, `local_clip_source_video_id` equal to `source_video_id`, `local_clip_selected_text_sha256` equal to `selected_text_sha256`, `local_clip_relative_path` as a workspace-relative portable path, `local_clip_file_size_bytes` as positive output file size, `local_clip_content_sha256` as output file sha256, `local_clip_begin_ms` / `local_clip_end_ms` equal to `selected_begin_ms` / `selected_end_ms`, `location_verified: true`, a non-empty workspace/source/segment/local-clip id, a SHA-256 hash of the selected text, `completed_closed_loop: true`, `workspace_output_written: true`, `public_library_written: false`, and finite search/detail/cut timings. Text attachments such as logs, JSON reports, CSV/TSV files, subtitles, and Markdown notes are also scanned for forbidden secret or private-transcript markers.

Windows screenshot attachments must be OS-specific. Keep Windows 10 screenshots under `screenshots/windows-10/` and Windows 11 screenshots under `screenshots/windows-11/`; the final gate rejects reused screenshot paths across environments.
