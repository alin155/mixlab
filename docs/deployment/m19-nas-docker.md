# M19 NAS Docker Deployment

## Deployment Goal

Run MixLab management services on the NAS while keeping the Windows cutter desktop app independent.

M19 moves only the management side to NAS Docker:

- `admin-web`: browser UI for the management side.
- `admin-api`: management API.
- `admin-worker`: recurring preprocessing and ready-publish worker.

The cutter desktop app and local `cutter-api` remain on the editor's Windows computer.

## NAS Folder Contract

- NAS shared folder: `共享文件夹/MixLab/PublicLibrary`
- Container path: `/data/PublicLibrary`
- Generated index pointer: `/data/PublicLibrary/.mixlab-library/indexes/source-transcript-index/current.json`

The user should not manually create `current.json`. It is generated by the worker after source videos have been processed and a ready index has been published.

## Services

| Service | Purpose | External Port |
| --- | --- | --- |
| `admin-web` | Management UI | `8080` by default |
| `admin-api` | Internal API for `admin-web` | Not exposed |
| `admin-worker` | Preprocess and publish ready source materials | Not exposed |

## First Deployment

Before copying files to the NAS, run the source-repository static deployment check:

```sh
npm run validate:nas-docker-compose-static
```

This validates the expected Compose services, GHCR images, public-library mount, worker command, `MIXLAB_PREPROCESS_COUNT_REFRESH_INTERVAL=25`, blank `.env.example` ASR key, and the internal-only boundary that `admin-api` and `admin-worker` do not publish host ports. It does not replace `docker compose config`, image pull/build checks, or real NAS target evidence.

1. Create the NAS folder `共享文件夹/MixLab/PublicLibrary`.
2. Copy `deploy/nas/mixlab/docker-compose.yml` to the NAS Docker app project folder.
3. Copy `deploy/nas/mixlab/.env.example` as `.env` in the same NAS Docker project folder.
4. Edit `.env`.
5. Set `PUBLIC_LIBRARY_HOST_PATH` to the NAS absolute path of `MixLab/PublicLibrary`.
6. Fill `DASHSCOPE_API_KEY`.
7. Log in to GHCR from the NAS Docker app if private image pull requires authentication.
8. Start the Compose project.
9. Open `http://NAS_IP:8080`.

## Environment Variables

| Name | Default | Meaning |
| --- | --- | --- |
| `PUBLIC_LIBRARY_HOST_PATH` | `/volume1/MixLab/PublicLibrary` | NAS host path mounted into `/data/PublicLibrary` |
| `MIXLAB_IMAGE_TAG` | `latest` | Docker image tag; use a Git SHA for rollback |
| `MIXLAB_ADMIN_WEB_PORT` | `8080` | NAS port exposed for the management UI |
| `MIXLAB_WORKER_POLL_INTERVAL_SECONDS` | `60` | Worker loop interval |
| `MIXLAB_ENABLE_LIBRARY_PREPROCESS_WORKER` | `1` | Enables source material preprocessing |
| `MIXLAB_ENABLE_READY_PUBLISH_WORKER` | `1` | Enables ready index publication |
| `MIXLAB_PREPROCESS_COUNT_REFRESH_INTERVAL` | `25` | Refreshes aggregate library counts every N successful preprocessing claims and once at worker-cycle end; failures still refresh immediately |
| `DASHSCOPE_API_KEY` | empty | Required for speech recognition preprocessing |

For large public libraries, keep `MIXLAB_PREPROCESS_FILE_IDENTITY_MODE=stat` and leave `MIXLAB_PREPROCESS_COUNT_REFRESH_INTERVAL=25` unless the NAS UI needs more frequent progress updates. Lower values make dashboard counts update more often but increase full-library metadata scans during the preprocessing worker cycle; higher values reduce NAS metadata churn during large imports.

## Verify Deployment

After the Compose project starts:

1. Confirm `admin-web` is reachable at `http://NAS_IP:8080`.
2. Confirm `admin-api` logs show the server listening on `0.0.0.0:3889`.
3. Confirm `admin-worker` logs show `admin-worker-loop-started`.
4. Add or approve a source video for preprocessing.
5. Confirm worker output appears under `.mixlab-library/`.
6. Confirm `current.json` exists after ready publication.

## Final Acceptance Evidence

After the NAS deployment and SMB tests are complete, encode the evidence in one JSON file and run:

```sh
npm run template:nas-evidence -- docs/acceptance/evidence/nas-acc-009.json
npm run validate:nas-evidence -- docs/acceptance/evidence/nas-acc-009.json
```

The template command creates an intentionally failing draft with every deployment, SMB, multi-user, and evidence-file slot. Replace every empty string, `false` value, and zero count with real NAS target evidence before running the validator. The NAS validator is an ACC-009 preflight; it also rejects attachment references that use absolute paths, backslashes, `..` traversal, or filename stems that do not match their evidence fields. Final delivery still requires the combined target gate after the Windows evidence file is also present:

```sh
npm run validate:target-evidence -- docs/acceptance/evidence/windows-acc-008.json docs/acceptance/evidence/nas-acc-009.json
```

On the NAS Docker host, the shell collector can fill deployment facts from the Compose logs, management URL, public-library folder, `current.json`, and supplied multi-user evidence fields:

Generate the standalone collector kit on the repository machine, then copy `dist/acceptance/mixlab-evidence-kit/nas/` to the NAS shell host:

```sh
npm run package:evidence-kit
```

Alternatively, download the `mixlab-target-evidence-kit` artifact from either the `Build Admin Docker Images` workflow or the `Target Evidence Kit` workflow, then copy its `nas/` folder to the NAS shell host.

The evidence kit's `captures/50-editor-report.json` is only a shape template. Fill it with the real 50+ editor NAS/SMB run before the collector copies it into `evidence/50-editor-report.json`; when Node is available on the NAS host, the collector rejects weak reports before copying them.

When returning NAS evidence to the repository, copy `nas-acc-009.json` and the generated `evidence/` folder together under `docs/acceptance/evidence/`. The NAS validator reads the relative screenshot, worker-log, `current.json`, SMB-permission, and 50-editor-report attachment paths and fails if only the JSON is copied back.

When running the repository-based 50-editor searchd smoke during NAS target validation, write the report directly into the collector captures folder. For final evidence, run it at the large public-library scale required by the final gate:

```sh
MIXLAB_SEARCHD_NAS_REHEARSAL_RUN_ROOT=/volume1/MixLab/AcceptanceRuns \
npm run smoke:searchd-nas-rehearsal
```

The rehearsal command defaults to `MIXLAB_SEARCHD_CONCURRENCY_VIDEOS=2000`, `MIXLAB_SEARCHD_CONCURRENCY_SEGMENTS_PER_VIDEO=24`, and `captures/50-editor-report.json`, and it rejects lower-scale overrides before running. The resulting report must still show `indexed_source_video_count >= 2000`, `indexed_transcript_segment_count >= 48000`, `searchd_health_source_video_count` equal to `indexed_source_video_count`, `searchd_health_segment_count` equal to `indexed_transcript_segment_count`, at least 50 unique `user_id` values, at least 50 unique `workspace_id` values, and per-editor search-hit-to-full-transcript-location proof before the collector copies it. The copied report must stay portable and must not include target-machine run root fields such as `library_root`, `workspace_root`, or `searchd_cache_root`.

Before collector copy, run the read-only report self-check from the copied NAS kit:

```sh
sh ./nas-50-editor-report-self-check.sh ./captures/50-editor-report.json
```

This catches raw report gaps before they are embedded in `nas-acc-009.json`. It does not replace the collector-side checks or final repository validators.

Then run the collector:

```sh
REPOSITORY_COMMIT_SHA=<40-char-commit-sha> \
EVIDENCE_KIT_WORKFLOW_RUN_URL=https://github.com/alin155/mixlab/actions/runs/RUN_ID \
ADMIN_WEB_URL=http://NAS_IP:8080 \
COMPOSE_PROJECT=mixlab \
IMAGE_TAG="$REPOSITORY_COMMIT_SHA" \
MIXLAB_PREPROCESS_COUNT_REFRESH_INTERVAL=25 \
PUBLIC_LIBRARY_HOST_PATH=/volume1/MixLab/PublicLibrary \
WINDOWS_UNC_PATH='\\NAS\MixLab\PublicLibrary' \
NAS_SHARED_FOLDER='共享文件夹/MixLab/PublicLibrary' \
READONLY_FROM_CUTTER=true \
SMB_PUBLIC_LIBRARY_NOT_WRITTEN_BY_CUTTERS=true \
ADMIN_SOURCE_VIDEOS_SOURCE_PATH_VISIBLE=true \
ADMIN_SOURCE_VIDEOS_FIRST_ID=V000001 \
ADMIN_SOURCE_VIDEOS_FIRST_ID_VISIBLE=true \
ADMIN_SOURCE_VIDEOS_LOADED_COUNT_BEFORE=100 \
ADMIN_SOURCE_VIDEOS_LOADED_COUNT_AFTER=200 \
ADMIN_SOURCE_VIDEOS_LOAD_MORE_INCREASED=true \
ADMIN_SOURCE_VIDEOS_QUERY='房产' \
ADMIN_SOURCE_VIDEOS_QUERY_RESULT_ID=V000001 \
ADMIN_SOURCE_VIDEOS_QUERY_API_RESPONSE_OBSERVED=true \
ADMIN_SOURCE_VIDEOS_QUERY_RESULT_VISIBLE=true \
ADMIN_SOURCE_VIDEOS_READY_FILTER_SELECTED=true \
ADMIN_SOURCE_VIDEOS_READY_FILTER_API_RESPONSE_OBSERVED=true \
ADMIN_SOURCE_VIDEOS_READY_FILTER_VISIBLE_STATUS_COUNT=100 \
ADMIN_SOURCE_VIDEOS_READY_FILTER_ALL_VISIBLE_ROWS_READY=true \
EVIDENCE_DIR=./captures \
sh ./nas-acc-009-collector.sh
```

Before leaving the NAS shell host, run the read-only local self-check from the copied kit:

```sh
sh ./nas-evidence-self-check.sh ./nas-acc-009.json
```

The self-check reports remaining draft placeholders, missing referenced attachments, fake or undersized screenshots, and obvious 50-editor report gaps while the target environment is still available. It does not replace the repository validators.

When Node is available on the collector host, the collector derives `editor_session_count`, `all_searches_passed`, `all_cuts_written_to_local_workspaces`, `public_library_not_written_by_cutters`, and `no_cross_workspace_outputs` from `captures/50-editor-report.json`. If Node or the report is unavailable, set `EDITOR_SESSION_COUNT=50`, `ALL_SEARCHES_PASSED=true`, `ALL_CUTS_WRITTEN_TO_LOCAL_WORKSPACES=true`, `MULTI_USER_PUBLIC_LIBRARY_NOT_WRITTEN_BY_CUTTERS=true`, and `NO_CROSS_WORKSPACE_OUTPUTS=true` from the observed target run before invoking the collector.

Use the `ADMIN_SOURCE_VIDEOS_*` variables only after opening the deployed NAS admin source-videos page against the real public source path. The completed `admin_source_videos_ui` evidence must prove that the source path is visible, a concrete source video id like `V000001` is rendered, load-more increases the loaded count, a keyword query observes the API response and rendered result, and the ready filter observes the API response with only ready rows. If these values were already filled in `nas-acc-009.json`, the collector preserves them when the matching environment variables are not supplied.

If Docker logs are available, the collector also captures the last 200 `admin-worker` log lines into `evidence/worker.log` when no explicit worker log file is supplied. That worker log must include `count_refresh_interval` matching `deployment.preprocess_count_refresh_interval`, so capture it after a preprocessing worker startup line has appeared. If Docker logs or `curl` are unavailable on the NAS shell, set the corresponding override variables (`ADMIN_WEB_REACHABLE`, `ADMIN_API_LISTENING`, `ADMIN_WORKER_LOOP_STARTED`, `WORKER_OUTPUT_CREATED`, `CURRENT_JSON_CREATED`) from the observed target evidence and provide `WORKER_LOG_EXCERPT` or `EVIDENCE_DIR/worker.log` manually.
Supplied NAS attachments must be non-empty. Screenshot files are rejected when their `.png`, `.jpg`, `.jpeg`, or `.webp` extension does not match the file signature; the NAS preflight and final combined gate reject missing, fake, undersized, or semantically swapped attachments, including screenshots smaller than 640x360 and attachment filenames that do not match their fields (`admin-web`, `current-json`, `smb-permissions`, `worker`, and `50-editor-report`). Text/log/report attachments are rejected if they contain ASR keys, signed URLs, private transcript text, pasted search text, or non-redacted bearer tokens. When Node or Python 3 is available, `.json` attachments are parsed before copying.
Set `REPOSITORY_COMMIT_SHA` to the 40-character commit that produced the NAS Docker images and `mixlab-target-evidence-kit` artifact. Keep `MIXLAB_PREPROCESS_COUNT_REFRESH_INTERVAL` aligned with the deployed NAS `.env`; the collector records it as `deployment.preprocess_count_refresh_interval`, and the final gate rejects missing or non-positive values. When the kit is downloaded from the `Build Admin Docker Images` workflow, the provenance fields and `deployment.image_tag` are already prefilled in `nas-acc-009.json`; the collector preserves those prefilled values if the matching environment variables are not provided again on the NAS shell host. The final combined gate rejects missing provenance, rejects workflow run URLs outside the official `https://github.com/alin155/mixlab` delivery repository, rejects NAS evidence when `deployment.image_tag` does not match that commit, and rejects paired Windows/NAS evidence that does not reference the same repository commit.

The validator requires:

- the 40-character repository commit SHA, `mixlab-target-evidence-kit` artifact name, evidence-kit workflow run URL, NAS `deployment.image_tag` matching that commit, and `deployment.preprocess_count_refresh_interval` from the deployed worker configuration;
- admin web reachability, admin API listen proof, admin worker loop proof, worker output, generated `current.json`, and worker log `count_refresh_interval` matching `deployment.preprocess_count_refresh_interval`;
- `admin_source_videos_ui` proof that the deployed NAS admin source-videos UI shows the real source path, visible source video id, load-more row growth, query API response plus rendered query result, and ready-filter API response with all visible rows ready;
- SMB public-library folder proof, Windows UNC path proof, cutter read-only behavior, and no cutter writes into the public library;
- at least 50 editor sessions with successful search/cut behavior across at least 50 distinct `source_video_id` values and at least 5 distinct `search_query` values, with `search_query_count` and `search_queries` matching the per-editor queries, `search_index_version` equal to the top-level searchd index version, `searchd_health_index_version` equal to `search_index_version`, `searchd_health_source_video_count` equal to `indexed_source_video_count`, `searchd_health_segment_count` equal to `indexed_transcript_segment_count`, search-result source/segment ids matching the full-transcript selected segment, `search_result_rank` inside `search_result_group_count`, `search_result_group_count` at least `search_result_limit`, `search_result_limit` at least 50, `full_transcript_source_video_id` equal to `source_video_id`, `full_transcript_segment_count >= 4`, `full_transcript_char_count` greater than `selected_text_char_count`, `full_transcript_text_sha256`, integer `search_result_begin_char` / `search_result_end_char` values falling inside the full-transcript character offsets, `search_result_text_sha256` equal to `sha256(search_query)`, integer `selected_text_begin_char` / `selected_text_end_char` values inside the full transcript that include the search hit range, `selected_text_char_count` equal to selected range length, `local_clip_source_video_id` equal to `source_video_id`, `local_clip_selected_text_sha256` equal to `selected_text_sha256`, `local_clip_relative_path` as a workspace-relative portable path, `local_clip_file_size_bytes` as positive output file size, `local_clip_content_sha256` as output file sha256, `local_clip_begin_ms` / `local_clip_end_ms` equal to `selected_begin_ms` / `selected_end_ms`, `location_verified: true`, local-workspace output isolation, no cross-editor workspace leakage, `indexed_source_video_count >= 2000`, and `indexed_transcript_segment_count >= 48000`;
- evidence file references for admin web, worker logs, `current.json`, SMB permissions, and the 50-editor search/cut report with matching attachment filenames and per-editor closed-loop rows. The collector copies these inputs into `evidence/` beside `nas-acc-009.json`; the JSON should not contain NAS absolute paths.

## Update

1. Pull the latest images in the NAS Docker app.
2. Restart the Compose project.
3. Confirm `admin-worker` logs show worker cycle output.
4. Confirm the management UI still opens.

## Rollback

1. Pick a known Git SHA image tag from GHCR.
2. Set `MIXLAB_IMAGE_TAG` to that SHA in `.env`.
3. Restart the Compose project.

## What M19 Does Not Move

M19 does not move the cutter desktop app or local `cutter-api` to NAS.

The editor's Windows app still handles local cutting, local output directories, and local editor-facing operations. NAS Docker only covers management-side public library preprocessing and management UI/API.
